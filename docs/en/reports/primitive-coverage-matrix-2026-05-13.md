# EvoUI Primitive Coverage Matrix

Date: 2026-05-13

This report compares the current evo-ui primitive surface, documentation
coverage and consumer alignment across `sArticles`, `sSeo`, `sLang`,
`sSettings`, `sTask`, `dIssues` and `dDocs`.

Percentages are practical contract-fit estimates from a static scan of package
`views`, `config`, `src`, `css` and `js` files. They are not PHPUnit coverage
percentages. `N/A` means the module does not currently use that primitive as a
primary surface.

Expanded component documentation is implemented English-first. Localized
translations are deferred to dIssues task `evo-ui-docs-049`.

## Main Findings

1. `sArticles` and `dIssues` are the strongest references. Use them as donor
   patterns for table/modal/workspace work, while keeping their domain logic out
   of evo-ui.
2. `sSeo` is close, but any manager-dashboard CSS should become EvoUI card/table
   primitive behavior. Local CSS should be only for site/content output.
3. `sLang` is split: the module table is good, the legacy resource-tab bridge is
   the remaining weak boundary.
4. `sSettings` is the best source for DnD behavior, but it also exposes the most
   local UI drift. The target is to consume EvoUI DnD/form primitives instead of
   owning row layout, option rows and dirty-state glue locally.
5. `sTask` uses EvoUI tables/dashboard internally but still has the largest shell
   and asset drift: legacy/CDN/local assets must be removed from manager UI.
6. `dDocs` should remain a documented UX exception for now. Promote tree/viewer
   primitives only after its API stabilizes.
